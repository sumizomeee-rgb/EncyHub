## 1. 背景与问题

### 1.1 问题描述

当前投屏功能在设备切换时会强制停止，用户需要手动重新启动。上一个开发尝试实现"自动重连"功能，但由于 scrcpy-server 的视频流 TCP 连接无法在 WebSocket 断开后恢复，导致了无限重连循环的 bug。

### 1.2 当前行为分析

| 场景 | 当前行为 | 用户感受 |
|------|----------|----------|
| 切换设备 | 投屏被静默停止 | 困惑：为什么投屏消失了？ |
| 切回原设备 | 需要手动重启投屏 | 繁琐：多一步操作 |
| 设备断开 | 投屏停止，无提示 | 不明确：不知道发生了什么 |

### 1.3 技术限制

scrcpy-server 的视频流通过 TCP 连接传输，当 WebSocket 断开后：
- TCP 视频流可能已经到达 EOF
- 无法在不重启 scrcpy-server 的情况下恢复视频流
- 因此"自动重连"方案不可行

## 2. 方案设计

### 方案 A：设备切换提示 + 快速恢复（推荐）

#### 2.1.1 核心设计原则

1. **明确告知**：切换设备时明确告知用户投屏已停止
2. **快速恢复**：提供便捷的方式恢复之前的投屏
3. **状态可见**：在 UI 上清晰展示当前投屏状态

#### 2.1.2 详细交互设计

**场景 A：当前正在投屏，用户点击其他设备**

```
用户操作：点击设备 B
系统行为：
  1. 自动停止设备 A 的投屏（调用后端 stop API）
  2. 显示 toast 提示："已停止设备 A 的投屏"
  3. 切换到设备 B
  4. 设备 B 的投屏面板显示"启动投屏"按钮
```

**场景 B：当前正在投屏，用户点击同一设备**

```
用户操作：点击设备 A（当前已选中且正在投屏）
系统行为：无变化，投屏继续
```

#### 2.1.3 投屏面板状态指示

在投屏面板的折叠头部增加设备信息显示：

```

│    正在投屏: Samsung Galaxy S21          │

│ ▶ 🖥️ 投屏控制                             │

└─────────────────────────────────────────┘
```

#### 2.1.4 设备卡片上的投屏指示

在设备列表中，显示哪个设备正在投屏：

```

│ Samsung Galaxy S21      [USB] [投屏中●]  │

┌─────────────────────────────────────────┐

```

#### 2.1.5 Toast 提示消息设计

| 场景 | Toast 消息 |
|------|------------|
| 切换设备（投屏中） | "已停止 Samsung Galaxy S21 的投屏" |
| 启动投屏成功 | "投屏已连接" |
| 停止投屏 | "投屏已停止" |
| 投屏异常断开 | "投屏连接已断开" |

#### 2.1.6 技术实现要点

**需要修改的文件**：
- `frontend/src/pages/AdbMaster.jsx`

**关键代码变更**：

```jsx
// 1. 新增状态变量
const [scrcpyStreamingDeviceId, setScrcpyStreamingDeviceId] = useState(null)

// 2. 设备切换处理函数
const handleSelectDevice = async (device) => {
  // 如果正在投屏且切换到不同设备，先停止当前投屏
  if (scrcpyStreaming && selectedDevice && selectedDevice.hardware_id !== device.hardware_id) {
    const prevDeviceName = selectedDevice.nickname || selectedDevice.model || selectedDevice.hardware_id
    await handleStopScrcpy(true) // silent stop, 不显示 toast
    toast.info(`已停止 ${prevDeviceName} 的投屏`)
  }
  setSelectedDevice(device)
}

// 3. 启动投屏时记录设备 ID
const handleStartScrcpy = async () => {
  // ... 启动逻辑
  setScrcpyStreamingDeviceId(selectedDevice.hardware_id)
  // ...
}

// 4. 停止投屏时清除设备 ID
const handleStopScrcpy = async (silent = false) => {
  // ... 停止逻辑
  setScrcpyStreamingDeviceId(null)
  // ...
}
```

#### 2.1.7 优点

- 用户体验流畅，不打断工作流
- 状态清晰可见，不会困惑
- 实现简单，改动量小

#### 2.1.8 缺点

- 切换设备仍需重新启动投屏
- 无法保留之前的投屏状态

### 方案 B：切换时弹出确认框

#### 2.2.1 交互设计

当用户在投屏中切换设备时，弹出确认框：

```
┌─────────────────────────────────────────┐
│ ⚠️ 提示                                  │
│                                          │
│ 当前正在投屏 Samsung Galaxy S21，        │
│ 切换设备将停止投屏。                      │
│                                          │
│        [取消]    [停止并切换]             │
└─────────────────────────────────────────┘
```

#### 2.2.2 用户操作流程

```
用户操作：点击设备 B（投屏中）
系统行为：
  1. 弹出确认框
  2. 用户选择：
     - [取消]：关闭对话框，保持在设备 A，投屏继续
     - [停止并切换]：停止投屏，切换到设备 B
```

#### 2.2.3 技术实现要点

```jsx
// 确认框状态
const [showSwitchConfirm, setShowSwitchConfirm] = useState(false)
const [pendingDevice, setPendingDevice] = useState(null)

// 设备点击处理
const handleSelectDevice = (device) => {
  if (scrcpyStreaming && selectedDevice?.hardware_id !== device.hardware_id) {
    setPendingDevice(device)
    setShowSwitchConfirm(true)
  } else {
    setSelectedDevice(device)
  }
}

// 确认切换
const confirmSwitch = async () => {
  await handleStopScrcpy()
  setSelectedDevice(pendingDevice)
  setShowSwitchConfirm(false)
  setPendingDevice(null)
}
```

#### 2.2.4 优点

- 用户有明确的决策权
- 可以取消误操作

#### 2.2.5 缺点

- 增加了用户的操作步骤
- 打断工作流，体验不够流畅
- 频繁切换设备时会感到厌烦

### 方案 C：多设备同时投屏

#### 2.3.1 核心概念

允许每个设备独立投屏，切换设备时自动切换画面。每个设备维护自己的投屏会话。

#### 2.3.2 交互设计

```
设备列表：
┌─────────────────────────────────────────┐
│ Samsung Galaxy S21      [USB] [投屏中●]  │
│ abc123...                               │
└─────────────────────────────────────────┘

┌─────────────────────────────────────────┐
│ Google Pixel 6          [WiFi] [投屏中●] │  ← 也在投屏
│ def456...                               │
└─────────────────────────────────────────┘

投屏面板：
┌─────────────────────────────────────────┐
│ ▼ 🖥️ 投屏控制 - Samsung Galaxy S21       │
│    [投屏中]                              │
│    ┌─────────────────────┐              │
│    │     Canvas 画面      │              │
│    └─────────────────────┘              │
└─────────────────────────────────────────┘
```

#### 2.3.3 用户操作流程

```
用户操作：启动设备 A 投屏
系统行为：设备 A 开始投屏，显示画面

用户操作：切换到设备 B
系统行为：设备 A 投屏保持（后台），设备 B 投屏面板显示"启动投屏"

用户操作：启动设备 B 投屏
系统行为：设备 B 开始投屏，显示画面

用户操作：切回设备 A
系统行为：设备 A 的投屏画面自动恢复显示（无需重新启动）
```

#### 2.3.4 技术实现要点

**前端状态管理**：

```jsx

// 当前选中设备的投屏状态
const currentSession = scrcpySessions[selectedDevice?.hardware_id] || null
```

**后端会话管理**：

```python
# 已支持多设备会话（ScrcpyWebManager._sessions 是 dict）
# 需要确保每个设备有独立的会话
```

**Canvas 渲染**：

```jsx
// 切换设备时，需要：
// 1. 保持前一个设备的 WebSocket 连接
// 2. 暂停渲染到 Canvas（节省性能）
// 3. 切回时恢复渲染
```

#### 2.3.5 优点

- 最理想的用户体验
- 切换设备无需等待
- 可以同时监控多个设备

#### 2.3.6 缺点

- **技术复杂度高**：需要重构整个前端状态管理
- **资源消耗大**：
  - 多路 WebSocket 连接
  - 多路 VideoDecoder 实例
  - 浏览器内存压力增加
- **后端压力**：多设备同时推流
- **设备限制**：浏览器对 WebSocket 连接数有限制
- **可能需要额外优化**：
  - 后台设备降低帧率/分辨率
  - 连接池管理
  - 自动断开长时间不活跃的会话

#### 2.3.7 资源消耗估算

| 设备数量 | WebSocket 连接 | VideoDecoder | 内存估算 |
|----------|----------------|--------------|----------|
| 1 | 1 | 1 | ~50MB |
| 2 | 2 | 2 | ~100MB |
| 3 | 3 | 3 | ~150MB |
| 4+ | 4+ | 4+ | 可能出现性能问题 |

## 4. 推荐方案

**推荐方案 A：设备切换提示 + 快速恢复**

理由：
1. 实现简单，改动量小，风险低
2. 用户体验流畅，不会打断工作流
3. 资源消耗低，稳定性高
4. 为后续升级到方案 C 留有空间

如果用户对方案 C 有强烈需求，建议作为独立的功能迭代进行开发，需要：
1. 充分的性能测试
2. 用户设置选项（启用/禁用多设备投屏）
3. 设备数量限制（建议最多 2-3 个同时投屏）
