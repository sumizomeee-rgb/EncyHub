#### Step 3: 修改 handleStopScrcpy 函数

**位置**: 原函数位置（约第744行）

```jsx
// ======== 停止投屏 ========
const handleStopScrcpy = async (silent = false) => {
  if (!selectedDevice) return

  // 关闭 WS
  if (scrcpyWsRef.current) {
    scrcpyWsRef.current.close()
    scrcpyWsRef.current = null
  }
  // 关闭解码器
  if (scrcpyDecoderRef.current && scrcpyDecoderRef.current.state !== 'closed') {
    scrcpyDecoderRef.current.close()
  }
  scrcpyDecoderRef.current = null
  setScrcpyStreaming(false)
  setScrcpyStreamingDeviceId(null) // 新增：清除投屏设备ID
  setScrcpyMeta(null)

  try {
    const res = await fetch(
      `/api/adb_master/devices/${selectedDevice.hardware_id}/scrcpy/stop`,
      { method: 'POST' }
    )
    const data = await res.json()
    if (!silent) {
      toast.success(data.message || '投屏已停止')
    }
  } catch (err) {
    if (!silent) {
      toast.error('停止失败: ' + err.message)
    }
  }
  setScrcpyStatus({ running: false })
}
```

#### Step 8: 更新现有的 selectedDevice 变化监听

**位置**: 现有 useEffect（第148-162行）

当前代码已在 `selectedDevice` 变化时清理前端资源，但需要补充：

```jsx
// 投屏: 设备切换时清理 WebSocket 和解码器
useEffect(() => {
  return () => {
    if (scrcpyWsRef.current) {
      scrcpyWsRef.current.close()
      scrcpyWsRef.current = null
    }
    if (scrcpyDecoderRef.current && scrcpyDecoderRef.current.state !== 'closed') {
      scrcpyDecoderRef.current.close()
      scrcpyDecoderRef.current = null
    }
    setScrcpyStreaming(false)
    setScrcpyStreamingDeviceId(null) // 新增：清除投屏设备ID
    setScrcpyMeta(null)
    // 注意：这里不显示 toast，因为设备切换时 toast 在 handleSelectDevice 中显示
    // 设备断开时 toast 在 WebSocket onclose 中显示
  }
}, [selectedDevice])
```

**重要说明**: 此 useEffect 在 `selectedDevice` 变化时触发清理，包括：
- 用户手动切换设备（已在 handleSelectDevice 中处理）
- 设备断开连接被 `fetchDevices` 清除

两种场景的 toast 显示策略不同，需要在各自位置处理。

## 7. 苏格拉底自检

### Q1: 为什么不选择方案B（确认框）？
**A**: 方案B增加用户操作步骤，打断工作流。方案A的自动停止+提示更流畅，用户实际操作中切换设备时通常期望停止当前投屏。

### Q2: 为什么不选择方案C（多设备同时投屏）？
**A**: 方案C技术复杂度高，资源消耗大（多路WebSocket+VideoDecoder），且浏览器WebSocket连接数有限制。作为首期优化，方案A性价比最高。

### Q3: 为什么需要 `scrcpyStreamingDeviceId` 状态？
**A**: 当前只有 `scrcpyStreaming` 布尔值，无法区分"正在投屏的是哪个设备"。新增此状态用于：
1. 设备卡片上显示正确的"投屏中"标签
2. 处理 WebSocket onclose 时判断是否需要显示提示

### Q4: 为什么 `handleStopScrcpy` 需要 `silent` 参数？
**A**: 区分两种场景：
- 用户主动停止：显示"投屏已停止"
- 设备切换时自动停止：显示"已停止 XXX 的投屏"

### Q5: 后端真的不需要修改吗？
**A**: 是的。后端 `ScrcpyWebManager` 已经：
1. 使用 Dict 存储会话，支持多设备
2. WebSocket 断开时自动清理会话
3. stop API 已正确实现

### Q6: 如果切换设备时 stop API 调用失败怎么办？
**A**: 前端已先关闭 WebSocket，投屏画面已停止。即使后端清理失败，用户体验不受影响。后端有超时清理机制。

### Q7: 状态同步是否有风险？
**A**: 风险低。所有状态更新都在 React 组件内，单向数据流。唯一需要注意 WebSocket onclose 时更新状态顺序。

### Q8: 设备断开时投屏状态如何处理？
**A**: 现有代码已有处理机制：
1. `fetchDevices` 每3秒轮询，发现设备消失会调用 `setSelectedDevice(null)`
2. `useEffect([selectedDevice])` 的清理函数会被触发
3. 清理函数会关闭 WebSocket 和解码器，清除 `scrcpyStreaming` 状态

**补充处理**：在清理函数中也清除 `scrcpyStreamingDeviceId`。

### Q9: 为什么不在 useEffect 清理中调用后端 stop API？
**A**:
1. 设备已断开，后端调用可能失败
2. 后端 WebSocket 断开时会自动清理会话
3. 如果需要调用，需要用 ref 保存设备ID，增加复杂度，收益不大

### Q10: 设备断开时如何显示提示？
**A**:
1. 设备断开 → `fetchDevices` 清除 `selectedDevice`
2. useEffect 清理 → 关闭 WebSocket
3. WebSocket onclose 触发 → 显示 "投屏连接已断开"

这是最自然的触发点，因为 WebSocket 断开意味着投屏真的结束了。
