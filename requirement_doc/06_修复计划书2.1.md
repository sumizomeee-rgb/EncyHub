# EncyHub 修复计划书 2.1 — 用户反馈修复

> 基于 2.0 修复后的用户实测反馈，逐项深度分析与修复方案。

## 一、ADB Master 问题

### 1.1 [P0] 推送功能应为路径输入而非文件上传

**用户反馈**: 推送功能应该提供给用户填文件夹路径，推送的是这个文件夹路径下的所有内容。

**原项目行为** (AdbMaster `ui_components.py:1330-1362`):
- Push 区域有 **两个文本输入框**（HistoryInput 组件）：
  - `push_local`: 本地路径（默认值为设备的 `Local_Sync_Area` 目录）
  - `push_remote`: 远程路径（默认 `/sdcard/Download/`）
- 用户手动输入本地文件夹/文件路径
- 旁边有 "Open Folder" 按钮（打开文件管理器浏览）
- ADB `push` 命令原生支持文件夹递归推送
- 路径历史按设备隔离存储

**EncyHub 现状**:
- 使用浏览器 `<input type="file">` 文件选择器
- 只能选单个文件，不支持文件夹
- 文件先上传到服务器 temp 目录，再 push 到设备
- 这是 Web 架构的根本限制 — 浏览器无法直接访问本地文件系统路径

**修复方案**:
将 Push 改为 **路径输入模式**（与原项目一致）：
- 移除 `<input type="file">` 文件选择器
- 改为文本输入框，用户填写服务器可访问的本地路径
- 后端直接从该路径 push 到设备（无需上传中转）
- 添加路径历史下拉（复用 Pull 的历史组件模式）
- 后端 Push API 改为接受 `local_path` + `remote_path` JSON 参数

**涉及文件**:
- `frontend/src/pages/AdbMaster.jsx` — Push UI 区域重写
- `tools/adb_master/main.py:227-256` — Push API 改为接受路径参数

### 1.2 [P0] Logcat 点击后显示"连接失败"

**用户反馈**: Logcat 点击后显示 `Logcat 连接失败`。

**根因分析** — WebSocket 代理链路排查:

前端连接路径: `ws://host/api/adb_master/devices/{hw_id}/logcat`

链路: 前端 WS → Hub 代理 (`hub_core/api.py:155`) → AdbMaster 子进程 WS

**可能失败点**:

1. **Hub WS 代理使用 `websockets.connect()`** (`api.py:181`)
   - 目标 URL: `ws://127.0.0.1:{tool.port}/devices/{hw_id}/logcat`
   - 如果 AdbMaster 子进程未启动或端口不可达，`websockets.connect()` 抛异常
   - 异常被 `except Exception: pass` 吞掉 (`api.py:219`)
   - 前端 WebSocket 被 accept 后立即关闭 → 触发 `ws.onerror`

2. **AdbMaster 端设备验证失败** (`main.py:191-194`)
   - 如果 `hw_id` 找不到设备或设备离线，发送 `{"error": "..."}` 后关闭
   - 但前端 `ws.onmessage` 不处理 JSON 错误消息，只追加文本到 logcat 数组

3. **ADB 子进程启动失败**
   - `start_logcat` 内部 `asyncio.create_task(capture_logcat())` 是 fire-and-forget
   - 如果 ADB 不存在或 serial 无效，子进程静默失败
   - WebSocket 保持连接但永远不会收到数据

4. **`on_line` 回调中 `asyncio.create_task(ws.send_text(line))` 异常被吞**
   - 如果 Hub 代理的上游 WS 已断开，`send_text` 失败但无人知道

**修复方案**:
- Hub WS 代理: 在 `websockets.connect()` 失败时，向前端发送错误消息再关闭
- AdbMaster logcat handler: 添加 try-catch 包裹 `start_logcat`，失败时发送错误
- 前端: `ws.onmessage` 检查是否为 JSON 错误消息，如果是则 toast 显示
- 前端: 增加更详细的错误日志（区分 onerror vs onclose）

**涉及文件**:
- `hub_core/api.py:155-227` — WS 代理错误处理
- `tools/adb_master/main.py:178-221` — Logcat WS handler
- `frontend/src/pages/AdbMaster.jsx:136-175` — toggleLogcat

## 二、GM Console 问题

### 2.1 [P1] GM 按钮 h-24 太高，需要降低

**用户反馈**: GM 按钮太高了，再改矮一些。

**现状**: 所有 GM 网格项使用 `h-24`（96px），对于短命令名称来说过高。

**修复方案**:
- 将 `h-24` 改为 `h-16`（64px）
- 所有类型统一: Btn、SubBox、Toggle、Input、CustomGM

**涉及文件**:
- `frontend/src/pages/GmConsole.jsx` — 所有 GM 网格项的 className

---

### 2.2 [P1] 缩放滑块视觉问题

**用户反馈**: 缩放滑块拉到底看上去还没拉到底，显示有问题。

**根因分析**:
HTML `<input type="range">` 的默认样式在不同浏览器下表现不一致。当 `min=1` 时，滑块轨道可能因为 thumb 宽度导致视觉上看起来没有到达最左端。

**修复方案**:
- 给 range input 添加自定义 CSS 样式，确保 thumb 在两端时视觉上贴边
- 或者在滑块旁显示当前值的数字标签（已有 `{gridCols}` 显示），让用户明确知道当前值
- 增加 range input 的宽度从 `w-20` 到 `w-28`，让滑动范围更大更精确

**涉及文件**:
- `frontend/src/pages/GmConsole.jsx` — range input 样式
- `frontend/src/index.css` — 可选: 添加 range 自定义样式

### 2.3 [P0] 游戏连接成功但 Web 无反应

**用户反馈**: 游戏里面显示连接成功，但是 web 完全没反应。

**根因分析** — WebSocket 事件链路排查:

前端连接路径: `ws://host/api/gm_console/ws/events`

链路: 前端 WS → Hub WS 代理 → GmConsole 子进程 `/ws/events`

**事件流**:
1. 游戏客户端 TCP 连接到 GmConsole 的 12581 端口
2. `server_mgr._handle_client()` 被调用，创建 Client 对象
3. `on_update()` 回调触发 → `asyncio.create_task(broadcast_event(...))`
4. `broadcast_event()` 遍历 `ws_connections` 发送 JSON
5. 但 `ws_connections` 存储的是 **GmConsole 子进程内部的 WebSocket 对象**
6. 前端通过 Hub 代理连接，Hub 代理使用 `websockets` 库连接到 GmConsole
7. Hub 代理的上游 WS 连接应该在 `ws_connections` 中

**关键问题**: Hub WS 代理 (`api.py:155-227`) 的双向转发逻辑:
- `forward_to_client()`: 从上游 WS 读取消息 → 发送到前端 WS
- `forward_to_upstream()`: 从前端 WS 读取消息 → 发送到上游 WS
- 使用 `asyncio.wait(FIRST_COMPLETED)` — 任一方断开则结束

**可能失败点**:
1. Hub 代理 `websockets.connect()` 连接到 GmConsole 子进程失败（端口不可达）
2. GmConsole 子进程的 `/ws/events` 端点正常工作，但 Hub 代理的转发有问题
3. `broadcast_event()` 中的 bare `except:` 吞掉了所有错误
4. 前端 WS 连接成功（`ws.onopen` 触发），但后续事件消息未到达

**调试方案**:
- 在 `broadcast_event()` 中添加日志打印，确认事件是否被发送
- 在 Hub WS 代理的 `forward_to_client()` 中添加日志，确认消息是否被转发
- 前端添加 WS 连接状态指示器（连接中/已连接/断开）
- 检查 GmConsole 子进程是否正常启动（查看 Hub 日志）

**修复方案**:
- `broadcast_event()`: 将 bare `except:` 改为 `except Exception as e:` 并打印日志
- Hub WS 代理: 添加连接失败的错误日志
- 前端: 添加 WS 连接状态指示（在顶部栏显示连接状态）
- 前端: HTTP fallback 轮询间隔从 10s 缩短到 3s（作为临时保底）

**涉及文件**:
- `tools/gm_console/main.py:33-39` — broadcast_event 错误处理
- `hub_core/api.py:155-227` — WS 代理日志
- `frontend/src/pages/GmConsole.jsx:96-168` — WS 连接 + 状态指示

## 三、苏格拉底自检

### Q1: Push 改为路径输入后，安全性如何保证？
**A**: 用户输入的是服务器本地路径，不经过浏览器上传。需要验证路径存在性（`os.path.exists`），但不需要做路径遍历防护，因为这是本地工具而非公网服务。

### Q2: Logcat 连接失败的根因是否确定？
**A**: 不完全确定。可能是 Hub WS 代理连接上游失败，也可能是设备验证失败。需要添加日志来定位。修复方案覆盖了所有可能的失败点，确保每个环节都有错误反馈。

### Q3: GM Console WS 事件链路是否有根本性架构问题？
**A**: 架构本身没问题（Hub 代理双向转发 + GmConsole 内部广播）。问题可能在于错误处理太弱（bare except 吞错误）和缺少日志。也可能是 GmConsole 子进程未正常启动。需要先添加日志确认。

### Q4: h-24 改为 h-16 是否会导致 Toggle/Input 类型内容溢出？
**A**: Toggle 类型包含名称 + 开关按钮，h-16 (64px) 足够。Input 类型包含名称 + 输入框 + 确认按钮，可能偏紧。可以考虑 Input 类型使用 `h-20` (80px) 而其他类型用 `h-16`。或者统一用 `h-18`（72px，需要自定义）。

### Q5: 滑块视觉问题是否值得投入自定义 CSS？
**A**: 增加宽度 (`w-20` → `w-28`) 是最简单的修复，能显著改善体验。自定义 range 样式是可选的增强。
