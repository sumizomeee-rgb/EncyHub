# EncyHub 调查分析报告 — 启动与部署问题

> 基于用户反馈"修复后毫无变化"的深度调查

---

## 一、核心结论

**所有修复代码已正确写入磁盘，但用户始终在访问旧进程。**

根因是 `start.bat` 缺少旧进程清理机制，导致：
1. 旧 Hub 进程占据 9524 端口不释放
2. 新 Hub 进程启动失败（ERROR 10048 端口冲突）
3. 用户浏览器访问的始终是旧进程 → 看到的是修复前的代码

---

## 二、ERROR 10048 端口冲突 — 根因链

### 2.1 复现路径

```
用户执行 start.bat
  ↓
[步骤 4/5] npm run build → 成功，覆盖 frontend/dist/ 文件
  ↓
[步骤 5/5] .venv\Scripts\python.exe main.py
  ↓
新 Hub 进程启动 → lifespan() 触发
  ↓
process_manager.startup_restore() → 启动 3 个工具子进程（新 PID、新端口）
  ↓
uvicorn 尝试 bind 0.0.0.0:9524
  ↓
❌ ERROR 10048: 端口已被旧进程占用
  ↓
新 Hub 进程退出 → lifespan shutdown → 杀死刚启动的 3 个工具子进程
  ↓
旧 Hub 进程仍在运行，旧工具子进程仍在运行
  ↓
用户访问 localhost:9524 → 连接到旧进程 → 看到旧代码
```

### 2.2 为什么旧进程不会自动退出

- `start.bat` 没有任何 `taskkill` 或端口检查逻辑
- 用户关闭 cmd 窗口时，如果 Hub 进程是后台运行或 `pause` 后直接关窗口，Python 进程不一定被杀死
- Windows 下 `signal.SIGINT` handler 只在 Ctrl+C 时触发，关闭窗口走的是 `SIGBREAK` 或直接 `TerminateProcess`
- 工具子进程使用 `CREATE_NO_WINDOW` 标志，即使父进程死亡也不会自动退出

### 2.3 registry.json 状态污染

当前 `data/registry.json` 内容：
```json
{
  "adb_master": { "enabled": true, "port": 11030, "pid": 42828 },
  "gm_console": { "enabled": true, "port": 6948, "pid": 38880 }
}
```

这些是旧进程的 PID 和端口。虽然 `Registry._load()` 会清除 `port` 和 `pid`（第 66-67 行），但新进程启动后又写入了新值，然后新进程退出时又清除了。旧进程的内存中仍保留旧的 PID/端口映射。

## 三、"工具显示已断开但能刷新" — 分析

### 3.1 可能场景

每次执行 `start.bat`，会发生：
1. 新 Hub 启动 → `startup_restore()` 启动新工具子进程
2. 新 Hub 绑定端口失败 → `shutdown_all()` 杀死新工具子进程
3. 旧 Hub 的工具子进程未受影响，继续运行

但 `registry.json` 被新进程覆写了（新端口/PID → 然后清除为 null）。

旧 Hub 进程的内存中 registry 不受文件变化影响（只在启动时读取一次），所以旧 Hub 仍然知道旧工具的端口，代理请求仍然能到达旧工具子进程。

### 3.2 "偶尔刷新变成空白界面"

`npm run build` 执行时：
1. Vite 先清空 `frontend/dist/` 目录
2. 重新生成新文件

在清空到重新生成的短暂窗口内（约 1-3 秒），旧 Hub 的 StaticFiles 中间件读取磁盘时找不到文件 → 返回 404 → 白屏。

构建完成后，新文件写入磁盘，旧 Hub 的 StaticFiles 从磁盘读取新文件 → 页面恢复。

**但关键问题**：旧 Hub 的 Python 后端代码仍然是旧版本（内存中加载的），所以：
- ✅ 前端 JS/CSS 变更可能生效（从磁盘读取）
- ❌ 后端 API 变更不会生效（内存中的旧代码）
- ❌ Hub WS 代理变更不会生效
- ❌ 工具子进程代码变更不会生效

这解释了为什么用户看到"毫无变化"——即使前端文件更新了，后端 API 行为没变，前端的新逻辑调用新 API 会失败或走旧路径。

## 四、Logcat 连接失败 — 深度分析

### 4.1 WS 链路

```
前端 WS → Hub WS 代理 (api.py:155) → AdbMaster 子进程 WS (main.py:178)
```

### 4.2 可能失败点

**A. Hub WS 代理连接上游失败**
- `websockets.connect(target_url)` 在 `api.py:181`
- 如果 AdbMaster 子进程端口不可达，抛异常
- 旧代码中异常被 `except Exception: pass` 吞掉（第 219 行）
- 前端 WS 已被 accept，但上游连接失败 → 立即进入 finally → 关闭前端 WS
- 前端触发 `ws.onerror` → 显示"Logcat 连接失败"

**B. AdbMaster 设备验证失败**
- `main.py:191`: 如果 `hw_id` 找不到设备，发送 `{"error": "设备不存在或离线"}`
- 旧前端不解析 JSON 错误消息，只追加文本到 logcat 数组

**C. ADB 子进程启动失败**
- `adb_manager.py:510`: `asyncio.create_subprocess_exec` 启动 `adb logcat`
- 如果 ADB 路径错误或设备 serial 无效，子进程立即退出
- `capture_logcat()` 是 fire-and-forget task，失败无人知道

**D. `on_line` 回调中的 `asyncio.create_task` 问题**
- `main.py:200`: `asyncio.create_task(ws.send_text(line))`
- 这在 logcat capture 的 async task 内部调用，event loop 是活跃的，理论上可行
- 但如果 Hub 代理的上游 WS 已断开，`send_text` 会抛异常
- 异常被 bare `except:` 吞掉，连接从池中移除

### 4.3 最可能的根因

**Hub WS 代理连接上游失败**是最可能的原因。因为：
- 旧代码 `except Exception: pass` 完全吞掉错误
- 前端 WS 被 accept 后立即关闭 → 触发 onerror
- 用户看到的就是"Logcat 连接失败"

修复代码已经添加了错误消息发送和日志打印，但因为旧进程仍在运行，修复未生效。

## 五、GM Console "游戏连接成功但 Web 无反应" — 分析

### 5.1 事件链路

```
游戏客户端 TCP → GmConsole 子进程 TCP Server (server_mgr.py)
  ↓ on_update() 回调
asyncio.create_task(broadcast_event({...}))
  ↓
broadcast_event() 遍历 ws_connections 发送 JSON
  ↓
Hub WS 代理转发到前端
```

### 5.2 可能失败点

1. `broadcast_event()` 的 bare `except:` 吞掉所有错误（已修复但未生效）
2. Hub WS 代理的双向转发可能在某个环节断开
3. 前端 WS 连接可能未成功建立（同 Logcat 的 Hub 代理问题）

### 5.3 关键观察

如果前端 HTTP fallback 轮询（每 10s）也没有数据更新，说明问题不仅在 WS，HTTP API 也可能有问题。需要确认：
- `/api/gm_console/clients` HTTP 接口是否返回了连接的客户端
- 如果 HTTP 也没返回，说明 GmConsole 子进程的 TCP server 可能没有正确注册客户端

## 六、修复方案

### 6.1 [P0-紧急] start.bat 添加旧进程清理

这是所有问题的根因。必须在启动新进程前杀死旧进程。

```batch
:: 在启动服务器之前添加
echo [0/5] Cleaning up old processes...
for /f "tokens=5" %%a in ('netstat -aon ^| findstr ":9524 " ^| findstr "LISTENING"') do (
    echo       Killing old process on port 9524 (PID=%%a)
    taskkill /F /PID %%a >nul 2>&1
)
:: 等待端口释放
timeout /t 2 /nobreak >nul
```

同时在 `main.py` 中添加启动前端口检查：

```python
def kill_port_holder(port):
    """启动前杀死占用端口的旧进程"""
    import psutil
    for conn in psutil.net_connections():
        if conn.laddr.port == port and conn.status == 'LISTEN':
            try:
                psutil.Process(conn.pid).kill()
                print(f"[EncyHub] 已杀死占用端口 {port} 的旧进程 (PID={conn.pid})")
            except:
                pass
```

### 6.2 [P0] 工具子进程孤儿清理

`startup_restore()` 应该在启动新工具前，检查并杀死旧的工具子进程：

```python
async def startup_restore(self):
    for tool_id, tool in registry.get_all().items():
        if tool.enabled:
            # 先清理可能存在的旧进程
            self._cleanup_stale_process(tool_id)
            success, msg = await self.start_tool(tool_id)
```

### 6.3 [P1] 优雅关闭改进

- 在 `main.py` 中注册 `atexit` handler 确保工具子进程被清理
- 在 Windows 下注册 `SIGBREAK` handler（关闭窗口时触发）
- 工具子进程不使用 `CREATE_NO_WINDOW`，改为跟随父进程的 Job Object

### 6.4 [P1] Logcat 连接失败的额外调试

即使修复了启动问题，Logcat 仍可能因为以下原因失败：
- ADB 路径未正确配置（`path_utils.get_adb_path()` 需要验证）
- 设备 serial 在 logcat 启动时已变化（WiFi 重连导致 serial 变化）

建议在 AdbMaster 子进程日志中添加更多调试信息。

## 七、验证步骤

用户在修复 start.bat 后，应按以下步骤验证：

1. 打开任务管理器，手动杀死所有 `python.exe` 进程
2. 运行 `start.bat`
3. 确认无 ERROR 10048 错误
4. 确认控制台显示新的 PID 和端口
5. 访问 `http://localhost:9524`，Ctrl+F5 强制刷新
6. 验证 Push 区域显示路径输入框（而非文件选择）
7. 验证 GM Console 按钮高度降低
8. 验证 Logcat 连接

## 八、苏格拉底自检

### Q1: 为什么前端构建成功但用户看不到变化？
**A**: 前端文件确实更新到了 `frontend/dist/`，旧 Hub 进程的 StaticFiles 中间件从磁盘读取文件，理论上能看到新前端。但后端 API 行为未变（旧代码在内存中），新前端的逻辑依赖新 API（如 Push 的 JSON body），调用旧 API 会失败或行为不一致。加上浏览器可能缓存了旧的 JS bundle，导致用户看到的完全是旧界面。

### Q2: 仅修复 start.bat 是否足够？
**A**: 修复 start.bat 是必要条件，但不充分。还需要：
- 确保用户 Ctrl+F5 强制刷新浏览器
- 验证 ADB 路径配置正确（Logcat 依赖）
- 验证 GmConsole TCP server 正常工作

### Q3: 为什么不改用随机端口避免冲突？
**A**: Hub 端口 9524 是用户记忆的固定入口。改为随机端口会导致用户每次需要查看新端口号。正确做法是启动前清理旧进程。

### Q4: CREATE_NO_WINDOW 是否是孤儿进程的根因？
**A**: 部分是。`CREATE_NO_WINDOW` 使子进程脱离控制台，父进程退出时子进程不会收到信号。但即使不用这个标志，Windows 下关闭控制台窗口也不保证所有子进程被杀死。根本解决方案是使用 Windows Job Object 将子进程绑定到父进程。
