## 一、架构层面问题

### 问题 1：WebSocket 代理

**现状**：方案中只提到了 HTTP 请求代理，但 AdbMaster 的 Logcat、FlowSvn 的执行日志、GmConsole 的事件流都需要 WebSocket。

**问题**：FastAPI 的 `httpx` 代理方案不支持 WebSocket。

**解决方案**：
- 方案 A：主进程实现 WebSocket 代理（使用 `websockets` 库转发）
- 方案 B：前端直接连接子进程的 WebSocket（需要暴露子进程端口）
- **推荐方案 B**：前端通过 `/api/hub/tools` 获取工具端口，直接连接 `ws://host:port/ws/xxx`

### 问题 2：内网访问时子进程端口

**现状**：子进程绑定 `127.0.0.1:动态端口`，只有本机能访问。

**问题**：内网其他设备无法直接连接子进程的 WebSocket。

**解决方案**：
- 方案 A：子进程也绑定 `0.0.0.0`
- 方案 B：主进程实现 WebSocket 代理
- **推荐方案 A**：子进程绑定 `0.0.0.0`，简单直接

### 问题 3：子进程崩溃处理

**现状**：方案未提及子进程崩溃后的处理。

**问题**：子进程异常退出后，注册表状态不一致。

**解决方案**：
- 主进程定期检查子进程存活状态（心跳检测）
- 发现崩溃后更新注册表状态
- 可选：自动重启崩溃的工具

### 问题 4：GmConsole TCP 端口的内网访问

**现状**：GmConsole 的 TCP 服务器需要接收游戏客户端连接。

**问题**：如果 TCP 绑定 `127.0.0.1`，内网设备（如手机上的游戏）无法连接。

**解决方案**：TCP 服务器必须绑定 `0.0.0.0:12581`

## 二、工具迁移层面问题

### 问题 5：AdbMaster 的路径依赖

**现状**：原项目的 `adb_manager.py` 中 adb.exe 路径是相对于原项目目录的。

**问题**：迁移后路径会失效。

**解决方案**：
- 修改 `adb_manager.py`，使用配置文件或环境变量指定 adb.exe 路径
- 或者使用 `__file__` 相对路径定位

### 问题 6：FlowSvn 的任务计划路径

**现状**：FlowSvn 通过 `schtasks` 创建 Windows 任务计划，命令中包含脚本路径。

**问题**：迁移后，已创建的任务计划中的路径会失效。

**解决方案**：
- 迁移时需要更新或重建任务计划
- 或者在文档中说明：迁移后需要重新创建任务

### 问题 7：配置文件兼容性

**现状**：原项目的配置文件（config.json、custom_gm.json）有特定格式。

**问题**：新项目的数据目录结构不同，配置文件路径变化。

**解决方案**：
- 修改各工具的配置管理模块，支持自定义数据目录
- 通过环境变量 `DATA_DIR` 传递数据目录路径

## 三、前端层面问题

### 问题 8：API BaseURL

**现状**：前端需要知道后端 API 地址。

**问题**：内网访问时，localhost 不适用。

**解决方案**：
- 使用相对路径 `/api/xxx`（推荐）
- 或者前端动态获取 `window.location.origin`

### 问题 9：WebSocket 连接地址

**现状**：前端需要连接工具的 WebSocket。

**问题**：工具端口是动态的，前端如何知道？

**解决方案**：
- `/api/hub/tools` 返回工具的运行端口
- 前端根据端口构建 WebSocket URL：`ws://${host}:${port}/ws/xxx`

## 四、独立打包问题

### 问题 10：打包后的数据目录

**现状**：各工具独立打包成 EXE。

**问题**：打包后数据目录在哪里？是 EXE 同级目录还是固定路径？

**解决方案**：
- 打包后数据目录在 EXE 同级的 `data/` 目录
- 需要在 `build.py` 中处理路径逻辑

### 问题 11：独立运行 vs 聚合运行的配置

**现状**：工具既可以在 EncyHub 中运行，也可以独立打包运行。

**问题**：两种模式下数据目录不同，如何兼容？

**解决方案**：
- 通过环境变量 `DATA_DIR` 区分
- 聚合运行时：`DATA_DIR=EncyHub/data/adb_master`
- 独立运行时：`DATA_DIR=./data`（EXE 同级）
